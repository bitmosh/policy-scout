# Policy Scout — JSON API v1 Plan

## 1. Purpose

Planning document for a future stable Policy Scout JSON API v1 contract across CLI outputs, future Tauri adapter calls, and future local API surfaces. This is docs-only planning, not implementation.

**Core principle:** JSON must be agent/script-readable, redaction-safe, and preserve stable IDs and granular evaluation data.

---

## 2. Non-goals

This document does NOT:

- Change runtime behavior or CLI JSON output
- Add schema validation code
- Add Tauri/frontend/Rust files
- Add dependencies
- Fix JSON inconsistencies in this pass
- Implement JSON API v1
- Break existing JSON consumers without migration path

---

## 3. JSON Contract Doctrine

### Core Rules

- JSON must be agent/script-readable
- JSON must be redaction-safe
- JSON must preserve stable IDs where available
- JSON must preserve granular evaluation data where relevant
- JSON should distinguish error, warning, empty, null, and could_not_verify
- Redaction placeholders are protected evidence, not errors
- UI/adapters must not bypass policy, audit, sandbox, approval, or report boundaries
- Local-first durable state remains under Policy Scout data roots
- Future API envelopes should support Tauri without requiring Tauri to inspect raw SQLite or filesystem internals

### Status Semantics

- success: Command completed successfully
- risky: Command was evaluated as risky (SANDBOX_FIRST, REQUIRE_APPROVAL)
- denied: Command was denied (DENY, DENY_AND_ALERT)
- error: Command failed due to system error
- partial: Command completed but with partial verification (could_not_verify items present)

## 4. Current Source-of-Truth Status

Existing CLI JSON is the current operational source of truth. All current --json outputs work as-is for CLI users. This document proposes a future v1 contract for consistency across Tauri and local API surfaces.

**Current JSON review candidates:**

- run --json allowed path uses decision_id, not full decision
- run --json blocked path lacks risk_score
- eval run --json uses execution_time_ms
- Some commands may use different list/detail shapes
- report list created_at field presence varies by report generation path

These are future JSON API v1 review items, not bugs to patch now.

## 5. Proposed v1 Response Envelope

**Proposal only, not current behavior:**

```json
{
  "schema_version": 1,
  "command": "policy-scout ...",
  "ok": true,
  "status": "success|risky|denied|error|partial",
  "request_id": null,
  "evaluation_id": null,
  "decision_id": null,
  "audit_event_ids": [],
  "report_id": null,
  "data": {},
  "warnings": [],
  "errors": [],
  "could_not_verify": [],
  "redaction_applied": true
}
```

**Field meanings:**

- schema_version: JSON contract version for compatibility detection
- command: Full command string for audit traceability
- ok: Boolean success indicator (true if status is success or partial)
- status: High-level status classification
- request_id: Unique request identifier if applicable
- evaluation_id: Unique evaluation identifier if applicable
- decision_id: Unique decision identifier if applicable
- audit_event_ids: List of audit event IDs generated by this command
- report_id: Report ID if a Scout Report was generated
- data: Command-specific response data
- warnings: Non-critical warnings (e.g., package manager not found)
- errors: Critical errors that prevented completion
- could_not_verify: List of items that could not be verified
- redaction_applied: Boolean flag indicating redaction was applied

## 6. ID and Relationship Rules

### Canonical ID Prefixes

- req_: CommandRequest
- eval_: EvaluationPacket
- dec_: PolicyDecision
- evt_: AuditEvent
- sbx_: SandboxResult
- sweep_: SweepResult
- find_: Finding
- report_: ScoutReport
- appr_: ApprovalRequest

### ID Relationships

- request_id links all events for a single command request
- evaluation_id links parse, classification, risk, policy, and decision
- decision_id links decision to approval, sandbox, or execution
- event_id links audit events to request_id and evaluation_id
- report_id links Scout Reports to request_id, decision_id, sandbox_id, or sweep_id
- sandbox_id links sandbox results to request_id and decision_id
- sweep_id links sweep results to project or system context

### ID Stability

IDs must be stable across audit, report, and data store lookups. IDs should be generated once and preserved in all related records.

## 7. Redaction and Privacy Rules

### Redaction Placeholders

Redaction placeholders are protected evidence, not errors. Display as-is (e.g., `<redacted:possible_token>`, `<redacted:ssh_private_key>`).

### Redaction Metadata

- redaction_applied: Boolean flag indicating redaction was applied
- redaction_fields: Optional list of field names that were redacted (future)

### Privacy Rules

- Never display raw secrets in JSON output
- Never log raw secrets to console
- Never include secrets in error messages
- Treat all JSON responses as potentially containing redacted data
- Normalize home paths to ~ where practical
- Do not resolve paths to absolute paths for privacy

### Redaction-Safe Fields

Command strings, file paths, environment variable names, process command lines, and credential-adjacent content must be redacted before JSON output.

## 16. Error, Warning, and Partial-Verification Semantics

### Error Semantics

- error: Critical failure that prevented command completion
- Exit code 30 for CLI errors
- JSON should include errors array with error_code, error_message, error_type

### Warning Semantics

- warning: Non-critical issue that did not prevent completion
- Examples: package manager not found, optional feature unavailable
- JSON should include warnings array

### Partial-Verification Semantics

- could_not_verify: List of items that could not be verified
- Status should be "partial" when could_not_verify is non-empty
- Examples: package scripts that could not be parsed, files that could not be read
- UI should display could_not_verify items as gray indicators

### Empty vs Null vs Missing

- empty: Valid zero-length array or empty object
- null: Explicit null value (field exists but is null)
- missing: Field not present in JSON
- UI should handle all three cases gracefully

## 18. Compatibility and Migration Strategy

### Backward Compatibility

- Do not break existing JSON consumers without migration path
- Introduce schema_version field for compatibility detection
- Support both old and new field names during transition period if needed
- Document breaking changes in release notes

### Migration Path

1. Add schema_version field to all JSON outputs
2. Add new fields (evaluation_id, decision_id, finding_id, redaction_applied) while keeping old fields
3. Deprecate old field names after transition period
4. Remove old field names in major version bump

### Version Detection

Clients should check schema_version field to determine JSON contract version. If schema_version is missing, assume v0 (current behavior).

## 19. Tauri/Local Adapter Implications

### Adapter Requirements

- Tauri adapter should parse schema_version field
- Tauri adapter should handle missing fields gracefully (defensive coding)
- Tauri adapter should display redaction_applied prominently
- Tauri adapter should treat could_not_verify as partial status
- Tauri adapter should not inspect raw SQLite or filesystem internals

### Local API Implications

- Future local HTTP API should use same JSON contract as CLI
- Local API should support schema_version negotiation
- Local API should preserve all IDs and relationships
- Local API should not bypass policy, audit, sandbox, approval boundaries

### Error Handling

- Adapters should distinguish error, warning, and partial-verification
- Adapters should display errors prominently
- Adapters should display warnings as non-critical
- Adapters should display could_not_verify as gray indicators

- Should execution_time_ms be aliased to duration_ms or renamed?
- Should all time fields use duration_ms consistently?
- Should decision_id be included in all decision-related commands?

### ID Consistency

- Should all commands return request_id?
- Should evaluation_id be added to all decision-related commands?
- Should finding_id be added to all finding arrays?

### Redaction Metadata

- Should redaction_fields list be added to show which fields were redacted?
- Should redaction_applied be required or optional?
- How to handle redaction in nested objects (findings, events)?

### Error Response Format

- Should errors return structured JSON with error_code, error_message, error_type?
- Should error responses use the same envelope as success responses?
- How to handle errors in long-running commands (sweep, eval)?

### Pagination

- Should list commands support cursor-based pagination?
- Should all list commands support consistent pagination parameters?
- How to handle large datasets in Tauri adapter?

### Filtering

- Should filter parameters be standardized across list commands?
- Should audit list support --type, --severity, --date-range filters?

### Envelope Adoption
