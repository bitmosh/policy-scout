# Policy Scout — Glossary and Conventions

## 1. Purpose

This document defines shared terminology and naming conventions for Policy Scout.

Agents and developers should use this vocabulary consistently.

---

## 2. Project Name

```text
Policy Scout
```

One-liner:

```text
Policy Scout is a local-first safety harness for agent commands, package installs, and suspicious project activity.
```

Tagline:

```text
Policy Scout: light armor for fast agents.
```

---

## 3. Core Terms

### Actor

Anything requesting an action.

Examples:

- human
- agent
- IDE
- CLI
- CI
- unknown caller

### Command Request

A structured representation of a requested command.

### Policy Engine

The deterministic authority that decides what happens to a request.

### Policy Decision

The final action returned by the policy engine.

Allowed decisions:

```text
ALLOW
ALLOW_LOGGED
REQUIRE_APPROVAL
SANDBOX_FIRST
DENY
DENY_AND_ALERT
```

### Evaluation Packet

The complete granular record of parsing, classification, capabilities, context, registry hits, risk, policy, decision, execution, and findings.

### Scout Report

A human-readable and machine-readable report explaining a decision, sandbox result, sweep result, or incident.

### Sweep

A scan of project or local system signals for suspicious traces.

### Sandbox

A temporary review workspace where risky package installs can be analyzed before host mutation.

### Approval Queue

A local queue of risky actions waiting for human approval.

### Registry

A local data file defining command behavior, policies, suspicious patterns, or indicators.

### Finding

A suspicious or informative result from a sweep, sandbox, or evaluation.

### Evidence

A reference supporting a finding or decision.

Evidence should avoid exposing raw secrets.

## 5. Naming Conventions

### File Names

Use uppercase snake case for docs:

```text
PROJECT_SCOPE.md
POLICY_DESIGN.md
DATA_MODELS.md
```

Use lowercase snake case for Python files:

```text
command_classifier.py
risk_scorer.py
sqlite_store.py
```

### IDs

Use prefixed IDs:

```text
req_      request
eval_     evaluation
dec_      decision
risk_     risk score
appr_     approval
exec_     execution
sbx_      sandbox
sweep_    sweep
find_     finding
evt_      audit event
report_   report
```

### Registry IDs

Use lowercase dot notation:

```text
npm.install
package_installs_sandbox_first
network_execute_deny
credential_access_deny_and_alert
```

### Categories

Use lowercase snake case:

```text
package_install
network_execute
credential_adjacent
```

### Capabilities

Use dotted capability names:

```text
network.fetch
network.execute
filesystem.project_write
credential.access_possible
```

## 6. Decision Names

Decision names should be uppercase.

```text
ALLOW
ALLOW_LOGGED
REQUIRE_APPROVAL
SANDBOX_FIRST
DENY
DENY_AND_ALERT
```

Do not invent new decision names without updating:

- TAXONOMIES.md
- POLICY_DESIGN.md
- DATA_MODELS.md
- tests

## 10. Report Tone

Use calm, precise language.

Prefer:

```text
Policy Scout found a suspicious lifecycle script.
Review is recommended before host installation.
Credential exposure is possible.
Policy Scout could not verify network behavior.
```

Avoid unsupported claims:

```text
Your machine is infected.
Malware confirmed.
Credentials definitely stolen.
```

unless evidence is confirmed.

---

## 11. Redaction Conventions

Use placeholders:

```text
<redacted:possible_token>
<redacted:ssh_private_key>
<redacted:env_value>
```

Do not print raw secrets.

---

## 12. RPG / Armor Metaphor Terms

These terms are allowed in branding or explanation, but code should remain clear.

### Light Armor

Policy Scout's protection philosophy: strong enough to protect, light enough to preserve agent agility.

### Scout Report

A report generated by Policy Scout.

This term is official.

### Scout Harness

Optional phrase for the runtime safety harness.

### Bandit Armor

Fun/internal name for adaptive non-critical optimization inspired by bandit-style learning.

Do not use this as a safety authority.

### Policy Warden

Possible internal name for enforcement logic, but prefer `policy_engine` in code.

### Aegisweave / Sigilweave

Conceptual armor terms for branding. Not required in implementation.

## 14. Conventions for Agents

Agents should:

- use exact taxonomy names
- avoid inventing new categories casually
- update docs when adding terms
- add tests for new terms
- preserve redaction
- preserve local-first behavior

---

## 15. Glossary Doctrine

Shared language is part of the safety system.

If agents and humans use the same terms, the project stays coherent.

If names drift, policy logic drifts too.
